# Corrupt JPEG data 警告 原因特定手順書

`controller.py` 実行中に大量発生する
`Corrupt JPEG data: N extraneous bytes before marker 0xdN` の原因を特定するための手順。
**すべて🍓ラズパイ上で実施する。** 誰が実行しても同じ結果・同じ判断になるよう、
コマンドは全てコピペ可能な完成形で記載し、判定基準は数値で明示する。

> この警告の意味: カメラがMJPG(圧縮JPEG)で送ってきた映像データに余分なバイトが
> 混ざっており、デコーダが読み飛ばしたという通知。フレーム自体は取得できている
> （read失敗にはなっていない）。問題は「なぜデータが汚れるのか」。

---

## 0. 共通ルール（全テストで毎回守ること）

1. **テスト前に必ず残存プロセスを確認する:**

   ```bash
   ps aux | grep python3
   ```

   `grep` 自身以外の行があれば:

   ```bash
   sudo pkill -f python3
   ```

2. **テストは60秒固定・640x480・条件は1つだけ変える**（スクリプトが強制する）

3. **stderrのリダイレクトを絶対に忘れない。**
   `Corrupt JPEG` 警告はCライブラリがstderr直接出力するため、
   リダイレクトし忘れると「0件」に見えてしまう（最頻出の誤操作）。

4. 各テスト後、必ず以下の2つの数値を記録表に記入する:
   - スクリプトが最後に表示する「総フレーム」
   - `grep -c "Corrupt JPEG" /tmp/stderr.log` の出力（警告数）

5. **破損率 = 警告数 ÷ 総フレーム（2台の場合は合計）× 100** を計算して記入する。

## 記録表（これをコピーして使う）

```
| # | テスト内容            | カメラ | FPS | 総フレーム | 警告数 | 破損率% | 備考 |
|---|----------------------|--------|-----|-----------|--------|---------|------|
| 1 | 基準測定(2台同時)     | 0+2    | 10  |           |        |         |      |
| 2a| video0単体           | 0      | 10  |           |        |         |      |
| 2b| video2単体           | 2      | 10  |           |        |         |      |
| 3 | 2台同時・低FPS       | 0+2    | 5   |           |        |         |      |
| 4 | 電力確認(コマンドのみ)| -      | -   |           |        |         |      |
| 5 | 物理入れ替え(必要時)  |        |     |           |        |         |      |
```

---

## ステップ1: 頻度の定量化（基準測定）

**目的**: 全フレーム中何%が破損しているかを数値化する。深刻度の判断と、
以降のテストの比較基準を作る。

```bash
cd ~/advance_software_engnering/AdvSoftwareG5
python3 scripts/camera_diagnosis.py --cameras 0 2 --fps 10 2> /tmp/stderr.log
grep -c "Corrupt JPEG" /tmp/stderr.log
```

記録表の1行目に記入したら、破損率で判定:

| 破損率 | 判定 | 次の行き先 |
|--------|------|-----------|
| 1%未満 | 実害なしレベル（数%のフレームが軽く乱れるだけで認識への影響は軽微） | 診断終了。警告の表示抑制のみ検討 |
| 1%以上 | 原因特定を続行 | ステップ2へ |

---

## ステップ2: カメラ個体の切り分け

**目的**: どちらのカメラが警告源かを特定する。
⚠️ **両方のカメラは物理的に挿したまま**にする（物理条件を変えないため）。

```bash
# video0（UVC Camera 046d:081b）のみ
python3 scripts/camera_diagnosis.py --cameras 0 --fps 10 2> /tmp/stderr.log
grep -c "Corrupt JPEG" /tmp/stderr.log

# video2（C922 Pro Stream Webcam）のみ
python3 scripts/camera_diagnosis.py --cameras 2 --fps 10 2> /tmp/stderr.log
grep -c "Corrupt JPEG" /tmp/stderr.log
```

記録表 2a / 2b に記入したら判定:

| 結果 | 判定 | 次の行き先 |
|------|------|-----------|
| 片方だけ破損率1%以上 | その個体・ケーブル・ポート固有の問題 | **ステップ5へ直行** |
| 両方とも1%未満（単体では出ない） | 「2台同時」でのみ発生する競合 | ステップ3へ |
| 両方とも1%以上 | 全体に共通する要因（電力 or 設定） | ステップ4へ |

---

## ステップ3: 同時使用の影響確認（FPSを下げて比較）

**目的**: 転送量を半分にして改善するかを見ることで、帯域/転送競合が原因かを確定する。

```bash
python3 scripts/camera_diagnosis.py --cameras 0 2 --fps 5 2> /tmp/stderr.log
grep -c "Corrupt JPEG" /tmp/stderr.log
```

記録表3に記入し、**ステップ1（FPS10）の破損率と比較**して判定:

| 結果 | 判定 |
|------|------|
| 破損率がステップ1の半分以下に下がった | **帯域/転送競合が原因と確定。** 対策候補は「FPS恒久引き下げ」「解像度削減」（修正は原因確定の報告後に別途計画） |
| ほぼ変わらない | 転送量に依存しない要因 → ステップ4へ |

---

## ステップ4: USB電力の確認

**目的**: 電圧降下によるデータ化けの有無を確認する（帯域ではなく電力の問題の切り分け）。

```bash
vcgencmd get_throttled
```

出力の読み方（`throttled=0x` に続く値）:

| 値 | 意味 |
|----|------|
| `0x0` | 電圧・温度とも問題なし（過去も含めて） |
| `0x50000` | **過去に**電圧低下があった（今は回復） |
| `0x50005` | **現在**電圧低下中 |
| 上記以外 | 何らかの異常履歴あり。値をそのまま報告する |

続けて:

```bash
dmesg | grep -iE "under-voltage|over-current" | tail -5
```

判定:

| 結果 | 判定 |
|------|------|
| `0x0` かつ dmesg該当なし | 電力は問題なし → ステップ5へ |
| それ以外 | **電力不足が疑われる。** 対策は物理対応（5V/2.5A以上の電源に交換、またはカメラをセルフパワーUSBハブ経由に）。コード修正では直らない |

---

## ステップ5: 物理層の切り分け（最後の手段）

**目的**: ケーブル・ポート・カメラ個体の物理的問題を特定する。
1回に変えるのは1条件だけ。変更のたびに60秒テストを実行し記録する。

| 試行 | 変更内容 | 実行コマンド | 見るべきこと |
|------|---------|-------------|-------------|
| 5-1 | 問題のカメラを**別のUSBポート**に挿し替え | ステップ2と同じ単体テスト | 警告が消えるか（→元ポートの問題） |
| 5-2 | 2台の**ポートを入れ替え** | 単体テスト×2 | 警告がカメラに付いて回るか（→個体）、ポートに残るか（→ポート） |
| 5-3 | （可能なら）**別のUSBケーブル/カメラ**に交換 | 単体テスト | 警告が消えるか（→ケーブル/個体） |

⚠️ 挿し替えるとデバイス番号（video0/video2）が変わることがある。
テスト前に必ず確認する:

```bash
v4l2-ctl --list-devices
```

## 診断終了後

記録表の全行をそのまま報告する。その内容に基づいて原因を確定し、
対策（コード修正 or 物理対応）を別途計画する。

## 診断結果と結論（2026-07-16 実施）

### 記録表（実測値）

| # | テスト内容 | カメラ | FPS | 総フレーム | 警告数 | 破損率% | 備考 |
|---|----------------------|--------|-----|-----------|--------|---------|------|
| 1 | 基準測定(2台同時) | 0+2 | 10 | 1168 | 573 | 49.1 | read失敗は0 |
| 2a| video0単体 | 0 | 10 | 573 | 573 | **100** | 全フレーム破損 |
| 2b| video2単体 | 2 | 10 | 596 | 0 | **0** | C922は完全にクリーン |
| 4 | 電力確認 | - | - | - | - | - | throttled=0x0、dmesg該当なし |
| 5-1| video0を別ポートへ | 0 | 10 | 564 | 552 | 97.9 | ポート要因を除外 |
| 5-3| PC直結で撮影 | 0 | - | - | - | - | 正常撮影→個体故障を除外 |
| A | video0単体・YUYV | 0 | 10 | 570 | **0** | **0** | 警告消滅・フレームレート低下なし |
| B | 0=YUYV + 2=MJPG 同時 | 0+2 | 10 | 572+596 | **0** | **0** | 本番構成で解決を確認 |

補足: `v4l2-ctl --list-formats-ext` でMJPG 640x480@10fpsはネイティブ対応
（フォーマット不一致説は除外）。

### 結論

**video0（Logicool C310, 046d:081b）のMJPG転送と Raspberry Pi 3 のUSB
コントローラ(dwc_otg)の相性問題。** カメラ個体・ケーブル・ポート・電源は
すべて除外済み。C922は同条件で無影響。

### 対策（実装済み）

`app/config.py` の `NO_MJPG_CAMERA_INDEXES = {MONITOR_CAMERA_INDEX}` により、
video0のみMJPGを使わずYUYV(無圧縮)で開く（`app/controller.py` の
`_open_camera` が参照）。JPEGデコード自体が無くなるため原理的に再発しない。
controller.py 実機テストでも警告0・`select() timeout` 0を確認済み。

## よくある誤操作チェックリスト

- [ ] `2> /tmp/stderr.log` を付け忘れて警告が画面に流れている
      （画面に `Corrupt JPEG` が見えたらリダイレクト忘れ。やり直す）
- [ ] 前のテストのプロセスが残ったまま次を実行した（ルール1を毎回実施）
- [ ] `controller.py` や `web_server.py` 等、本体プログラムが裏で動いている
- [ ] テストごとにFPS/カメラ以外の条件も変えてしまった（1条件ずつ）
- [ ] `grep -c` の対象ファイルが前のテストのログのまま（毎回上書きされるので、
      テスト→grepを続けて実行する）
