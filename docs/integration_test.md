# 結合テスト手順書（AI認識モジュール）

明日の結合テストに向けた手順・想定結果・トラブル対策のまとめ。

---

## 1. 結合テストの目的

単体で動作確認済みの各モジュールを実際につなぎ、システム全体が設計どおりに動くことを確認する。

```
ラズパイ（カメラ撮影 → JPEG送信）
    ↓  POST /predict
Ubuntu サーバー（GPU推論: 野菜・硬貨・人間）
    ↓  JSON レスポンス
ラズパイ（個数集計 → 万引き判定 → 通知）
    ↓  POST /update_sales
サーバー（売上管理・Web UI 表示）
```

### 確認したいこと

| # | 確認項目 | 合格基準 |
|---|----------|----------|
| 1 | ラズパイ→サーバーの画像送信 | `/predict` が200を返す |
| 2 | 推論結果のJSON形式 | `detections` に class_name / confidence / bbox が入る |
| 3 | 野菜・硬貨・人間の同時検出 | 1リクエストで3種すべて返る |
| 4 | 連続送信時の安定性 | 3〜5FPSで数分間エラーなく動く |
| 5 | 他モジュールとの連携 | 検出結果を重量センサー・売上管理側が受け取れる |

## 2. テスト手順

### 事前準備（テスト開始前に済ませておく）

- [ ] サーバー側: 学習済みモデル `ai/runs/vegetables_v1/weights/best.pt` が存在すること
- [ ] サーバー側: `yolov8s.pt`（人間検出用）がダウンロード済みであること（初回起動時に自動DLされるが、当日ネットワークが不安定な場合に備え事前に一度サーバーを起動しておく）
- [ ] サーバーとラズパイが**同じネットワーク**にいること
- [ ] サーバーのIPアドレスを確認: `ip addr show`（例: 192.168.1.10）

### 手順1: サーバー起動（Ubuntu側）

```bash
cd /workspace   # またはリポジトリのルート
python app/web_server.py
```

`Starting minimal web server on port 8080...` が表示されれば起動成功。
モデルロードに数十秒かかるので、表示されるまで待つ。

### 手順2: 疎通確認（ラズパイ側から）

```bash
# まずステータス確認（軽い通信でネットワーク疎通を確認）
curl http://<サーバーIP>:8080/status
```

`{"sales_count": 0, ...}` が返ればネットワークはOK。

### 手順3: 静止画での推論テスト

```bash
# ラズパイまたは任意のPCから
python app/simulate_raspi.py --image <テスト画像> --server http://<サーバーIP>:8080
```

検出結果（class_name / confidence / bbox）が表示されれば合格。

### 手順4: カメラからの連続送信テスト

```bash
# ラズパイ実機
python app/main.py --server http://<サーバーIP>:8080

# またはWebカメラ付きPCで模擬
python app/simulate_raspi.py --camera 0 --server http://<サーバーIP>:8080
```

- 野菜・硬貨・人（自分が写り込む）を順にカメラに見せ、検出されることを確認
- 5分程度連続稼働させ、エラーやメモリ増加がないことを確認

### 手順5: 他モジュールとの結合

- 検出結果JSONを重量センサー担当のロジック（`check_purchase_or_theft`）に渡す
- `/update_sales` への通知が Web UI に反映されることを確認

## 3. 起きると考えられる問題と対策

### 通信系

| 問題 | 原因 | 対策 |
|------|------|------|
| `/status` すら繋がらない | 別ネットワーク・ファイアウォール | 同じWi-Fi/LANか確認。`sudo ufw allow 8080` で開放 |
| `Connection refused` | サーバー未起動・ポート違い | サーバーのターミナルで起動ログを確認 |
| タイムアウトする | 推論が遅い・初回モデルロード中 | 起動直後は1リクエスト目が遅い。2回目以降で判断 |
| DevContainer内のサーバーに外から繋がらない | ポートフォワーディング未設定 | `devcontainer.json` の `forwardPorts: [8080]` を確認。ダメならホスト側で直接 `python app/web_server.py` を実行 |

### 推論系

| 問題 | 原因 | 対策 |
|------|------|------|
| 誤検出・検出漏れが多い | 実環境（照明・角度）が学習データと違う | confidence閾値（`DEFAULT_CONF=0.25`）を上げる/下げる。結合テストでは「検出の正確さ」より「パイプラインが動くか」を優先 |
| 硬貨の種類を間違える | 照明・距離の影響 | カメラを硬貨に近づける・真上から撮る。当日は100円玉が最も精度が高い（テスト済み） |
| 人間が検出されない | 全身が写っていない | カメラから離れて上半身以上が写るようにする |
| GPUエラーで落ちる | ドライバ・CUDA問題 | `nvidia-smi` でGPU確認。ダメなら`device="cpu"`に切り替え（遅いが動く） |

### 運用系

| 問題 | 原因 | 対策 |
|------|------|------|
| FPSが出ない・遅延が大きい | ネットワーク帯域・推論詰まり | 送信FPSを下げる（`main.py` の `fps=10` → `3`）。画像サイズを640に縮小してから送る |
| 長時間でサーバーが重くなる | メモリリーク | 現状1リクエスト毎に完結する設計なので起きにくいが、起きたらサーバー再起動で復旧 |
| SSH切断でサーバーが止まる | フォアグラウンド実行 | `tmux new -s server` の中でサーバーを起動しておく |
| ラズパイのカメラが開けない | デバイス未接続・権限 | `ls /dev/video0` で存在確認。コンテナ実行時は `--device /dev/video0` が必要（`scripts/start-container.sh` が対応済み） |

### 想定外の事態に備えて

- **サーバーが完全に動かない場合の代替**: `simulate_raspi.py --image` で静止画テストに切り替え、パイプラインの動作だけでも実証する
- **ラズパイが動かない場合の代替**: PCの `simulate_raspi.py --camera 0` でラズパイの役割を代行
- **ネットワークが使えない場合の代替**: サーバー1台で `localhost` に対して送受信し、パイプライン自体の動作を見せる

## 4. 結合テストで得られるもの

- **実環境での精度の把握**: 学習データと実際のカメラ映像のギャップが数値でわかる → 再学習の要否判断材料
- **実効FPSの測定**: 「3〜5FPSで動くはず」の仮説を実測で検証できる
- **モジュール間インターフェースの検証**: JSON形式が他担当の実装と噛み合うか確認できる
- **残課題の洗い出し**: 発表までに直すべき点の優先順位付けができる
